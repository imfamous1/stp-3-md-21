# Git - система контроля версий

Git - распределенная система контроля версий.

## Что это дает?

### Контроль версий

Git сохраняет не просто файлы, а историю изменений каждого файла. Можно в любой момент вернуться к каждому этапу.

### Распределенная система

У каждого разработчика в команде есть полная копия всего проекта и всей его истории на своем компьютере. Это дает независимость и скорость.

**Пример работы:** Представьте, что над одним проектом работают несколько человек. Каждый делает свою копию, работает независимо. Когда готов поделиться, предлагает свою версию изменений ответственному лиду. Лид может принять эти правки либо отклонить с комментариями. В любой момент можно посмотреть, кто, что и когда менял.

## Основные понятия

### Repo (репозиторий)

Это папка, в которой Git начал отслеживать изменения. Внутри нее всегда есть скрытая папка `.git` - это и есть своего рода база данных всех версий.

**Инициализация:**
```bash
git init  # создает новый репозиторий в текущей папке
```

### Commit (коммит)

Снимок изменений в репозитории на конкретный момент времени. У каждого коммита есть:
- Уникальный хеш (например, `a1b2c3d`)
- Автор
- Дата
- Сообщение (важно - должно описывать, что было изменено)

**Создание коммита:**
```bash
git add .                    # добавляет изменения в staging area
git commit -m "Сообщение"     # создает коммит с описанием
```

### Branch (ветка)

Это "параллельная вселенная" вашего проекта. По умолчанию мы работаем в ветке `main`. Изолируем новую функциональность (фичи) или исправление багов, не ломая основную рабочую версию проекта.

**Работа с ветками:**
```bash
git branch                    # список всех веток
git branch название-ветки     # создать новую ветку
git checkout название-ветки   # переключиться на ветку
git checkout -b новая-ветка   # создать и переключиться сразу
```

### Remote (удаленный репозиторий, GitHub)

Общий "сборочный пункт" для всех копий проекта. Нужен для синхронизации работы команды.

**Связь с удаленным репозиторием:**
```bash
git remote add origin URL     # добавить удаленный репозиторий
git remote -v                 # посмотреть все удаленные репозитории
```

## Основные команды

### Status - статус изменений

Показывает, какие файлы изменены, добавлены или удалены.

```bash
git status
```

### Add - добавление в staging area

Подготавливает файлы к коммиту. Можно добавить конкретные файлы или все изменения.

```bash
git add файл.txt              # добавить конкретный файл
git add .                     # добавить все изменения
git add папка/                # добавить все из папки
```

### Commit - сохранение снимка

Создает коммит с описанием изменений.

```bash
git commit -m "Описание изменений"
git commit -am "Сообщение"    # добавить и закоммитить измененные файлы
```

### Push - отправка на сервер

Отправляет ваши коммиты в удаленный репозиторий (например, на GitHub).

```bash
git push origin main          # отправить в ветку main
git push                      # отправить в текущую ветку
```

### Pull - получение изменений

Скачивает и применяет изменения из удаленного репозитория.

```bash
git pull origin main          # получить изменения из ветки main
git pull                      # получить изменения из текущей ветки
```

### Clone - клонирование репозитория

Создает полную копию удаленного репозитория на вашем компьютере.

```bash
git clone URL-репозитория
```

### Checkout - переключение

Переключается между ветками или коммитами. Также может отменить изменения в файлах.

```bash
git checkout название-ветки   # переключиться на ветку
git checkout -b новая-ветка    # создать и переключиться
git checkout -- файл.txt       # отменить изменения в файле
```

### Merge - слияние веток

Объединяет изменения из одной ветки в другую. Обычно используется для добавления функциональности из feature-ветки в main.

```bash
git checkout main              # переключиться на основную ветку
git merge название-ветки       # влить изменения из другой ветки
```

**Процесс слияния:**
1. Работаете в отдельной ветке над фичей
2. Переключаетесь на `main`
3. Делаете `git merge` вашей ветки
4. Git объединяет изменения

### Log - история коммитов

Показывает историю коммитов в репозитории.

```bash
git log                       # полная история
git log --oneline             # краткая история (одна строка на коммит)
git log --graph               # с визуализацией веток
```

## Типичный workflow (рабочий процесс)

1. **Клонирование проекта:**
   ```bash
   git clone https://github.com/user/repo.git
   ```

2. **Создание ветки для новой фичи:**
   ```bash
   git checkout -b feature/new-button
   ```

3. **Внесение изменений и коммит:**
   ```bash
   git add .
   git commit -m "Добавлена новая кнопка"
   ```

4. **Отправка на сервер:**
   ```bash
   git push origin feature/new-button
   ```

5. **Слияние с main (через Pull Request на GitHub или локально):**
   ```bash
   git checkout main
   git pull origin main
   git merge feature/new-button
   git push origin main
   ```

## Полезные команды

- `git diff` - показать различия между файлами
- `git reset` - отменить изменения в staging area
- `git stash` - временно сохранить изменения без коммита
- `git tag` - создать метку для версии (например, v1.0.0)
- `.gitignore` - файл, который указывает Git, какие файлы игнорировать
